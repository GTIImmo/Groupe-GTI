# Note schema V1 demande diffusion

Date: 2026-03-31

## Objet

Documenter le schema V1 retenu pour la surcouche `demande de diffusion`.

Ce schema sert de base avant implementation UI Pauline / negociateur.

Attention :

- la `demande de diffusion` et la `console Diffusion` sont deux modules distincts
- la demande sert a ouvrir la diffusion d'un bien non diffusable
- la console `Diffusion` sert a piloter un bien deja diffusable

## Workflow retenu

Statut principal unique :

- `en_cours`
- `accepte`
- `refuse`

La relance n'est pas un statut principal.

## Tables retenues

### 1. `app_diffusion_request`

Table principale de la demande.

Elle stocke :

- le dossier / annonce / mandat concernes
- le createur de la demande
- le commentaire initial
- le statut courant
- la decision Pauline
- les donnees d'acceptation
- les donnees de refus
- les informations de relance

### 2. `app_diffusion_refusal_reason`

Catalogue parametrable des motifs types de refus.

Chaque motif embarque :

- un code
- un libelle
- un texte de refus
- un texte de relance
- un texte manager
- les delais de relance et d'escalade

### 3. `app_diffusion_request_event`

Historique des evenements metier.

Exemples :

- creation de la demande
- acceptation
- refus
- correction declaree
- relance envoyee
- manager notifie

Important Supabase :

- si `app_diffusion_request.id` est expose en `uuid` ou en `text` cote front, alors `app_diffusion_request_event.diffusion_request_id` doit etre du meme type
- il ne faut pas le creer en `bigint` si les demandes sont identifiees par UUID
- sinon les insertions d'evenements plantent avec une erreur `invalid input syntax for type bigint`

Patch recommande si la table vient d'etre creee et est vide :

```sql
drop table if exists public.app_diffusion_request_event;

create table public.app_diffusion_request_event (
  id bigint generated by default as identity primary key,
  diffusion_request_id text not null,
  event_type text not null,
  event_label text not null,
  event_at timestamptz not null default now(),
  actor_user_id text,
  actor_name text,
  actor_role text,
  payload_json jsonb,
  created_at timestamptz not null default now()
);

create index if not exists idx_app_diffusion_request_event_request
on public.app_diffusion_request_event(diffusion_request_id, event_at desc);
```

### 4. `app_diffusion_target`

Table de pilotage de la console `Diffusion`.

Elle stocke, pour chaque dossier et chaque passerelle :

- la cible souhaitee (`enabled` / `disabled`)
- l'identifiant technique `hektor_broadcast_id`
- le `portal_key`
- l'origine de la demande
- la derniere tentative d'application
- le dernier statut / la derniere erreur

Cette table ne remplace pas l'etat lu depuis Hektor.

Elle sert a stocker l'intention metier cote application.

## Regles

### A la creation

- insertion dans `app_diffusion_request`
- `status = 'en_cours'`
- ajout d'un event `request_created`

### A l'acceptation

- `status = 'accepte'`
- stocker `accepted_set_diffusable`
- stocker `accepted_default_portals_json`
- stocker la trace email
- ajouter un event `accepted`

Effets metier :

- le bien devient diffusable automatiquement
- les passerelles autorisees par defaut sont selectionnees automatiquement
- le pilotage fin des passerelles bascule ensuite dans la console `Diffusion`

### Au refus

- `status = 'refuse'`
- motif obligatoire
- stocker le message applique
- calculer `next_reminder_at`
- ajouter un event `refused`

### A la correction cote negociateur

- renseigner `corrected_at`
- stopper les relances automatiques
- ajouter un event `correction_declared`

## Motifs V1 retenus

- `elements_manquants`
- `mandat_non_valide`
- `bien_non_diffusable`
- `photos_non_conformes`
- `texte_annonce_incomplet`
- `bareme_honoraire_non_respecte`
- `validation_interne_requise`
- `correction_fiche_bien`
- `autre`

## Contraintes importantes

- une seule demande `en_cours` par dossier
- `refusal_reason_code` obligatoire si `status = 'refuse'`
- `accepted_default_portals_json` obligatoire si `status = 'accepte'`
- `decision_at` obligatoire si la demande est `accepte` ou `refuse`

Pour la console `Diffusion` :

- une ligne `app_diffusion_target` par couple `app_dossier_id + hektor_broadcast_id`
- la cible metier est independante de l'etat observe dans Hektor
- l'execution technique reste tracee dans `app_broadcast_action`

## Regle d'affichage UI

Sur la ligne annonce / mandat :

- si `diffusable <> 1` : afficher `Demande de diffusion`
- si `diffusable = 1` : afficher `Diffusion`

Le lien `Diffusion` doit etre visible pour :

- le negociateur
- Pauline
